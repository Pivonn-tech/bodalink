Contributing

Purpose

This document describes the contribution process for the BODALINK project.

All contributions should follow the engineering standards and project conventions.

---

Before Contributing

Contributors should:

- Read the project documentation.
- Review existing issues.
- Understand the project architecture.
- Create a feature branch before making changes.

---

Development Workflow

1. Update documentation if required.
2. Create a new branch.
3. Implement the change.
4. Test the implementation.
5. Commit using the project conventions.
6. Submit the change for review.

---

Branch Naming

Examples:

- feature/authentication
- feature/wallet
- feature/booking
- fix/login
- docs/api

---

Commit Messages

Preferred prefixes:

- feat
- fix
- docs
- refactor
- test
- chore
- ci

Each commit should represent one logical change.

---

Pull Requests

Every pull request should include:

- A summary of the change.
- Testing information.
- Documentation updates, if applicable.

---

Code Quality

Contributors should:

- Follow the project's coding standards.
- Write readable code.
- Avoid duplication.
- Keep changes focused.

---

Reporting Issues

Bug reports should include:

- Steps to reproduce.
- Expected behavior.
- Actual behavior.
- Environment information.

Feature requests should explain the problem and proposed solution.

---

Community Standards

Contributors are expected to communicate respectfully, provide constructive feedback, and help maintain a welcoming and professional development environment.
